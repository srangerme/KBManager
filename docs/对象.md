# 对象

本文定义系统中每类对象的职责、路径、最小结构和引用关系。

## 1. 通用字段

所有 Markdown 对象使用 YAML frontmatter。非 Markdown source 使用同名 `.meta.yml`。

通用必填字段：

```yml
id: object-20260520-001
type: object-type
title: 对象标题
status: active
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
```

通用规则：

- `id` 表示全局唯一的逻辑对象 ID。candidate 使用预分配的全局 knowledge ID；candidate 被 accept 后，原 candidate 文件被原子提升/迁移为正式 knowledge 文件，同一 ID 不得同时存在 candidate 文件和 knowledge 文件。
- `type` 必须与对象类型一致。
- `created` 不随内容更新而变化。
- `updated` 在对象事实内容变化时更新。
- 引用其他对象时使用对象 ID。读取时由 API 操作和期望对象类型决定查找 candidate 目录或 knowledge 目录；如果同一 ID 同时出现在 candidate 和 knowledge 中，视为一致性错误。
- 对象文件属于数据层事实来源，只能由 `kb.*` API 创建、修改、移动或归档。
- Claude Code 和 slash command 可以编排 API，但不得直接改对象文件来绕过状态机、review gate 或一致性规则。
- 单条 source、candidate、knowledge、knowledge base、note 都应有独立 Markdown 对象文件；PDF、图片等非 Markdown 资源作为原始资源或附件被对象引用。

## 2. Source

用途：保存原始材料或其元数据。

路径：

- Markdown：`data/raw/md/*.md`
- PDF：`data/raw/pdf/*.pdf` + `data/raw/pdf/*.meta.yml`
- URL/HTML：`data/raw/html/*.html` + `data/raw/html/*.meta.yml`

模板：

```yml
id: source-20260520-001
type: source
title: 原始材料标题
source_type: pdf
status: raw
path: data/raw/pdf/example.pdf
summary:
cleaned:
  path:
  generated_at:
  method:
authors: []
published_at:
imported_at: 2026-05-20T14:30:05
deprecated_at:
deprecated_reason:
tags: []
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
```

状态：

- `raw`
- `deprecated`
- `archived`

状态语义：

- `raw` 表示 source 原始资源仍作为事实来源保留；清洗结果不改变 source 的原始状态。
- `deprecated` 表示 source 内容已经过期、不再推荐作为新知识的依据，但仍保留为历史来源和引用链的一部分。
- `archived` 表示 source 不再活跃维护，但不必然代表内容错误或过期。
- `cleaned` 不是 source 的独立状态或独立对象，而是 source 对象中的派生字段。`cleaned.path` 可指向 `data/cleaned/` 下的清洗内容文件，必须能追溯到同一个 source。

关系：

- 可被 candidate 引用。
- 可被 note 绑定。
- 可维护 cleaned 派生字段。
- deprecated source 被新 candidate 引用时，必须在 review 中提示 user 确认。

## 3. Candidate

用途：保存 LLM 或 user 从 source、note 生成的候选知识。

路径：

- `candidates/pending/*.md`
- `candidates/rejected/*.md`
- `candidates/deferred/*.md`

模板：

```md
---
id: knowledge-20260520-001
type: candidate
title: 候选知识标题
status: pending
source_refs:
  - source-20260520-001
note_refs: []
suggested_tags: []
suggested_kb_ids: []
evidence:
  - source_id: source-20260520-001
    locator: page 1
    quote: 支撑该候选知识的原文摘录
relations: []
review:
  reviewed_by:
  reviewed_at:
  decision:
  reason:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## 候选知识

## 支撑材料

## LLM 生成说明
```

状态：

- `pending`
- `rejected`
- `deferred`

关系：

- 必须引用至少一个 source 或 note。
- `evidence` 每项必须引用上游 source 或 note，可使用 `source_id`、`object_id` 或 `id` 字段；必须包含非空 `locator`，并且必须包含 `quote`、`excerpt` 或 `snippet` 之一。
- `relations` 没有关联时必须是 `[]`；有关联时每项必须包含非空 `type` 和 `target`，且 `target` 必须指向已有正式 knowledge ID，不能指向 source、note、candidate 或本次新建 candidate。
- `id` 是预分配的全局知识 ID。
- accept 后，candidate 文件被原子提升/迁移为同 ID 的正式 knowledge 文件，不再保留 candidate 文件。
- merge 到已有 knowledge 时，最终知识使用目标 knowledge ID；来源 candidate 保留原 ID、状态为 `rejected`。
- rejected/deferred 仍保留 review 记录，且不会生成同 ID knowledge。

## 4. Knowledge

用途：保存通过 review 的正式原子知识。

路径：`knowledge/atomic/*.md`

模板：

```md
---
id: knowledge-20260520-001
type: knowledge
title: 正式知识标题
status: accepted
tags: []
source_refs:
  - source-20260520-001
note_refs: []
evidence:
  - source_id: source-20260520-001
    locator: page 1
    quote: 支撑该知识的原文摘录
kb_ids: []
relations:
  - type: supports
    target: knowledge-20260520-002
reviewed_by: user
reviewed_at: 2026-05-20T14:30:05
review_decision: accept
review_reason:
deprecated_at:
deprecated_reason:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## 知识正文

## 支撑材料

## 关系说明
```

状态：

- `draft`
- `accepted`
- `deprecated`

状态语义：

- `deprecated` 表示知识已经过期或不再推荐使用，但仍作为历史知识保留，可继续被检索、引用和审计。
- 被其他 knowledge 明确取代时，仍使用 `deprecated` 状态。
- `deprecated` 不等于 `rejected`。它曾经可以成立，只是后来过期或被新的事实、规则、实践替代。

关系：

- 必须能追溯到 source 或 candidate。
- `relations` 没有关联时使用 `[]`；有关联时每项必须包含非空 `type` 和 `target`，且 `target` 指向已有 knowledge ID。accept/merge 的 reviewed payload 即使没有关系也必须显式传 `relations: []`。
- accepted knowledge 使用来源 candidate 预分配的全局知识 ID；accept 是文件提升/迁移，不产生同 ID candidate 与 knowledge 共存。
- merge 生成的 knowledge 使用目标 knowledge ID，来源 candidate 的 ID 不成为正式 knowledge ID。
- 可属于多个 knowledge base。
- 可与其他 knowledge 建立关系。
- deprecated knowledge 仍保留对象文件和引用链用于审计；默认列表索引不展示 deprecated 对象，避免作为推荐知识继续使用。如需审计，应读取对象文件或通过 `/check` 的一致性问题定位。

## 5. Knowledge Base

用途：组织一组 knowledge，形成主题型或用途型知识库。

路径：`knowledge/bases/*.md`

派生索引路径：

- 全局 knowledgebase index：`indexes/kb-index.md`
- 单个 knowledgebase 的 knowledge index：`indexes/knowledgebase/<knowledgebase-id>-knowledge-index.md`

模板：

```md
---
id: kb-20260520-001-知识库标题
type: knowledge-base
title: 知识库标题
status: active
description: 知识库说明
acceptance_criteria: 知识准入规则
knowledge_ids:
  - knowledge-20260520-001
tags: []
reviewed_by: user
reviewed_at: 2026-05-20T14:30:05
review_decision: approve
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## 说明

## 知识列表
```

状态：

- `active`
- `archived`

关系：

- 引用多个 knowledge。
- 新建 knowledge base 未显式指定 ID 时，API 使用标题生成可读 ID，格式为 `kb-YYYYMMDD-001-<title-slug>`；旧格式 `kb-YYYYMMDD-001` 仍可作为已有对象或显式 ID 使用。
- knowledge 通过 `kb_ids` 反向引用 knowledge base。
- knowledge base 当前由用户通过 `/knowledgebase create` 创建；`kb.knowledge.accept` 和 `kb.knowledge.merge` 会根据最终 `kb_ids` 同步维护 `knowledge_ids`，但 `kb.*` API 不提供独立 add/remove 成员操作。
- 手工编辑对象文件后，`knowledge_ids` 与 knowledge `kb_ids` 仍可能出现不一致；`/check` 应提示用户修复，并以对象文件为事实来源重建索引。

## 6. Note

用途：保存个人笔记、临时想法、草稿。note 默认不是正式知识。

路径：

- `notes/inbox/*.md`
- `notes/bound/*.md`
- `notes/deprecated/*.md`
- `notes/archive/*.md`

模板：

```md
---
id: note-20260520-001
type: note
title: 笔记标题
status: inbox
bindings:
  - type: knowledge
    id: knowledge-20260520-001
tags: []
summary:
deprecated_at:
deprecated_reason:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## 笔记
```

状态：

- `inbox`
- `bound`
- `deprecated`
- `archived`

关系：

- 可绑定 source、candidate、knowledge、knowledge base。
- `bindings` 没有关联时使用 `[]`；有关联时每项形如 `{type: source|candidate|knowledge|knowledge-base, id: <object-id>}`，且 `id` 必须存在并与 `type` 匹配。
- note 本身不是 knowledge 的事实来源，除非经 review 转化。
- `deprecated` 表示 note 过时或不再推荐使用，不做物理删除。

## 7. Index

用途：保存从对象文件派生出来的定位视图。

路径：`indexes/*`

常见索引：

- `manifest.yml`
- `source-index.md`
- `knowledge-index.md`
- `tag-index.md`
- `relation-index.yml`
- `kb-index.md`
- `knowledgebase/*-knowledge-index.md`
- `note-index.md`
- `review-queue.md`

规则：

- 索引不是事实来源。
- 索引可由对象文件重建。
- 索引损坏时以对象文件为准。
- 索引不得包含对象文件中不存在的事实。
- 默认列表索引只展示当前可用对象，`deprecated` 对象从列表索引中隐藏；对象文件仍是审计事实来源。

## 8. System Prompt

用途：描述 LLM 参与命令和 API 时使用的 KBManager 系统提示词。用户侧不提供提示词文件。

KBManager 系统提示词：

- 路径：KBManager 本体中的 `system-prompts/*.md`。
- 归属：随 KBManager 发布，不属于用户工作区。
- 内容：角色定义、边界、行为规则、输出约束、review gate、错误处理。
- 限制：不得包含用户 source、note、knowledge、索引或私有输入。

模板：

```md
---
id: system-prompt-candidate-create-v1
type: system-prompt
title: Candidate Create Prompt
api: kb.candidate.create
version: 1
inputs:
  - upstream_object
  - existing_knowledge
outputs:
  - candidate_draft
  - evidence_refs
  - tag_suggestions
review_required: true
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## Role

## Input Variables

## Output Format

## Constraints

## Review Requirements
```

状态：

- `active`
- `deprecated`

关系：

- 系统提示词可被 Interface 命令和 `kb.*` API 的 `needs_llm` 请求引用。
- 被 API 运行记录使用版本。
- KBManager 系统提示词不存放具体业务输入。

## 11. 关系类型

建议关系类型：

- `supports`：支持。
- `contradicts`：冲突。
- `refines`：细化。
- `related_to`：相关。

关系由 `kb.*` API 写入对象 frontmatter，并可派生到 `indexes/relation-index.yml`。
