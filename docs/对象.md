# 对象

本文定义系统中每类对象的职责、路径、最小结构和引用关系。

## 1. 通用字段

所有 Markdown 对象使用 YAML frontmatter。非 Markdown source 使用同名 `.meta.yml`。

通用必填字段：

```yml
id: object-20260520-001
type: object-type
title: 对象标题
status: active
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
```

通用规则：

- `id` 表示全局唯一的逻辑对象 ID。candidate 使用预分配的全局 knowledge ID；candidate 被 accept 后，原 candidate 文件被原子提升/迁移为正式 knowledge 文件，同一 ID 不得同时存在 candidate 文件和 knowledge 文件。
- `type` 必须与对象类型一致。
- `created` 不随内容更新而变化。
- `updated` 在对象事实内容变化时更新。
- 引用其他对象时使用对象 ID。读取时由 API 操作和期望对象类型决定查找 candidate 目录或 knowledge 目录；如果同一 ID 同时出现在 candidate 和 knowledge 中，视为一致性错误。
- 对象文件属于数据层事实来源，只能由 `kb.*` API 创建、修改、移动或归档；clean 迁移在完整迁移计划获得用户整批确认后的迁移执行，以及用户显式要求的 `kbm-kb-outline` outline YAML 更新，是受控特许例外。
- Claude Code UI 和 `kbm-*` skills 可以编排 API，但不得直接改对象文件来绕过状态机、review gate 或一致性规则；clean 特许迁移和 outline YAML 更新也不得绕过各自的确认和边界要求。
- 单条 source、candidate、knowledge、knowledge base、note 都应有独立 Markdown 对象文件；PDF、图片等非 Markdown 资源作为原始资源或附件被对象引用。

## 2. Source

用途：保存原始材料或其元数据。

路径：

- Markdown：`data/raw/md/*.md`
- PDF：`data/raw/pdf/*.pdf` + `data/raw/pdf/*.meta.yml`

模板：

```yml
id: source-20260520-001
type: source
title: 原始材料标题
source_type: pdf
status: raw
path: data/raw/pdf/example.pdf
summary:
cleaned:
  path:
  generated_at:
  method:
deprecated_at:
deprecated_reason:
tags: []
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
```

状态：

- `raw`
- `deprecated`

状态语义：

- `raw` 表示 source 原始资源仍作为事实来源保留；清洗结果不改变 source 的原始状态。
- `deprecated` 表示 source 内容已经过期、不再推荐作为新知识的依据，但仍保留为历史来源和引用链的一部分。
- `cleaned` 不是 source 的独立状态或独立对象，而是 source 对象中的派生字段。`cleaned.path` 可指向 `data/cleaned/` 下的清洗内容文件，必须能追溯到同一个 source。

关系：

- 可被 candidate 引用。
- 可维护 cleaned 派生字段。
- deprecated source 被新 candidate 引用时，必须在 review 中提示 user 确认。

## 3. Candidate

用途：保存 LLM 或 user 从 source 生成的候选知识。

路径：

- `candidates/pending/*.md`
- `candidates/rejected/*.md`
- `candidates/deferred/*.md`

模板：

```md
---
id: knowledge-20260520-001
type: candidate
title: 候选知识标题
status: pending
bindto:
  - kb_id: kb-20260520-001-知识库标题
    outline_id: canonical
    node_id: section-1
    reason: 该候选知识适合绑定到该大纲节点的原因
outline_change_suggestions:
  - kb_id: kb-20260520-001-知识库标题
    reason: 该内容属于知识库 scope，但当前 outline 未覆盖
    suggested_change: 建议新增或调整的大纲节点
summary: 候选知识总结
evidence:
  - source_id: source-20260520-001
    locator: page 1
    quote: 支撑该候选知识的原文摘录
review:
  reviewed_at:
  decision:
  reason:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## Content

## 支撑材料

## LLM 生成说明
```

状态：

- `pending`
- `rejected`
- `deferred`

关系：

- 必须通过 `evidence` 引用至少一个 source。
- `evidence` 每项必须引用上游 source，可使用 `source_id`、`object_id` 或 `id` 字段；必须包含非空 `locator`，并且必须包含 `quote`、`excerpt` 或 `snippet` 之一。
- `summary` 是给人快速 review 的候选知识总结；`content` 是候选知识正文，存放在 Markdown body 中。
- `bindto` 表示候选知识建议绑定到哪些 knowledge base 的哪些 `outline` 节点；没有合适绑定时使用 `[]`。
- `outline_change_suggestions` 表示内容属于某个 knowledge base 的 `scope`，但现有 `outline` 需要扩展或调整才能覆盖；该字段只记录建议，不会自动修改 knowledge base。
- candidate 和正式 knowledge 都只使用 `evidence` 追溯来源。
- candidate 归属建议只使用 `bindto` 和 `outline_change_suggestions`。
- `id` 是预分配的全局知识 ID。
- accept 后，candidate 文件被原子提升/迁移为同 ID 的正式 knowledge 文件，不再保留 candidate 文件。
- merge 到已有 knowledge 时，最终知识使用目标 knowledge ID；来源 candidate 保留原 ID、状态为 `rejected`。
- rejected/deferred 仍保留 review 记录，且不会生成同 ID knowledge。

## 4. Knowledge

用途：保存通过 review 的正式原子知识。

路径：`knowledge/atomic/*.md`

模板：

```md
---
id: knowledge-20260520-001
type: knowledge
title: 正式知识标题
status: accepted
summary: 正式知识总结
evidence:
  - source_id: source-20260520-001
    locator: page 1
    quote: 支撑该知识的原文摘录
bindto:
  - kb_id: kb-20260520-001-知识库标题
    outline_id: canonical
    node_id: section-1
    reason: 该知识绑定到该节点的原因
reviewed_at: 2026-05-20T14:30:05
review_decision: accept
review_reason:
deprecated_at:
deprecated_reason:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## Content

## 支撑材料
```

状态：

- `accepted`
- `deprecated`

状态语义：

- `deprecated` 表示知识已经过期或不再推荐使用，但仍作为历史知识保留，可继续被检索、引用和审计。
- 被其他 knowledge 明确取代时，仍使用 `deprecated` 状态。
- `deprecated` 不等于 `rejected`。它曾经可以成立，只是后来过期或被新的事实、规则、实践替代。

关系：

- 必须能追溯到 source 或 candidate。
- `summary` 是给人和 LLM 快速理解的总结性描述；`content` 是正式知识正文，存放在 Markdown body 中。
- `bindto` 是 knowledge 归属和结构绑定的事实来源，表示该知识绑定到哪些 knowledge base 的哪些 outline 与节点；没有绑定时使用 `[]`。每项必须包含 `kb_id`、`outline_id`、`node_id` 和 `reason`。
- `evidence` 是 knowledge 追溯 source 的唯一结构化来源字段。
- knowledge 组织与归属只由 knowledge base outlines YAML 和 knowledge `bindto` 表达。
- accepted knowledge 使用来源 candidate 预分配的全局知识 ID；accept 是文件提升/迁移，不产生同 ID candidate 与 knowledge 共存。
- merge 生成的 knowledge 使用目标 knowledge ID，来源 candidate 的 ID 不成为正式 knowledge ID。
- 可通过 `bindto` 绑定到多个 knowledge base 或多个 outline 节点。
- deprecated knowledge 仍保留对象文件和引用链用于审计；默认列表索引不展示 deprecated 对象，避免作为推荐知识继续使用。如需审计，应读取对象文件或通过 check workflow 的一致性问题定位。

## 5. Knowledge Base

用途：组织一组 knowledge，形成主题型或用途型知识库。

路径：`knowledge/bases/*.md`

派生索引路径：

- 全局 knowledgebase index：`indexes/kb-index.md`
- 单个 knowledgebase 的 knowledge index：`indexes/knowledgebase/<knowledgebase-id>-knowledge-index.md`

模板：

```md
---
id: kb-20260520-001-知识库标题
type: knowledge-base
title: 知识库标题
status: active
description: 知识库说明
tags: []
scope:
  includes: []
  excludes: []
default_outline_id: canonical
outlines_file: knowledge/bases/kb-20260520-001-知识库标题-outlines.yml
outlines:
  - id: canonical
    title: Canonical
    status: active
reviewed_at:
review_decision:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## 说明

## Scope

## Outline

## 派生成员视图
```

完整 outline 树不写入 knowledgebase Markdown frontmatter，而是写入 `outlines_file` 指向的 YAML 文件：

```yml
kb_id: kb-20260520-001-知识库标题
default_outline_id: canonical
outlines:
  - id: canonical
    title: Canonical
    description: 主大纲
    status: active
    nodes:
      - id: section-1
        title: 第一节
        children: []
```

状态：

- `active`
- `archived`

关系：

- 通过 accepted knowledge 的 `bindto` 派生引用多个 knowledge；knowledge base 对象本身不维护成员 ID 列表。
- `description` 是知识库总结性的描述，主要给人看。
- `tags` 是关键词集合，用于检索和粗粒度归类。
- `scope` 定义哪些内容属于知识库、哪些内容不属于知识库，是 candidate 生成和 review 判断归属的主要依据。
- `outlines_file` 是完整 outline 树的事实来源；knowledgebase Markdown frontmatter 中的 `outlines` 只是轻量 manifest，用于快速展示和检查。outline 可以是很大且复杂的节点树或节点列表，但每个可绑定节点必须有稳定 ID。knowledge 通过 `bindto[].outline_id` 和 `bindto[].node_id` 绑定到具体 outline 节点。
- 新建 knowledge base 未显式指定 ID 时，API 使用标题生成可读 ID，格式为 `kb-YYYYMMDD-001-<title-slug>`；旧格式 `kb-YYYYMMDD-001` 仍可作为已有对象或显式 ID 使用。
- knowledgebase create workflow 由 `kb.knowledgebase.create` 使用 source-like input 返回 `needs_llm` 生成草案；resume 后等待用户 review，用户确认后同一 API 创建 active knowledgebase Markdown 和 outlines YAML。该 input 不创建 source，也不成为 evidence。
- `kb.knowledgebase.outline.create`、`kb.knowledgebase.outline.set_default` 和 `kb.knowledgebase.outline.archive` 是修改 outline 的受控 API，均需要 user review。
- `kb.knowledge.accept` 和 `kb.knowledge.merge` 只更新 knowledge 的 `bindto`，不会修改 knowledge base outline，也不会在 knowledge base 对象中写入成员列表。
- knowledge base membership 的唯一事实来源是 accepted knowledge 的 `bindto`；`indexes/knowledgebase/*-knowledge-index.md` 等成员列表只作为派生视图存在。
- 手工编辑对象文件后，check workflow 应校验 knowledge `bindto` 指向的 knowledge base、outline 和节点是否存在，并以对象文件为事实来源重建成员索引。

## 6. Note

用途：保存个人笔记、临时想法、草稿。note 默认不是正式知识。

路径：

- `notes/active/*.md`
- `notes/deprecated/*.md`

模板：

```md
---
id: note-20260520-001
type: note
title: 笔记标题
status: active
deprecated_at:
deprecated_reason:
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## 笔记
```

状态：

- `active`
- `deprecated`

关系：

- note 不绑定 source、knowledge 或 knowledge base，也不维护对象引用关系。
- note 本身不是 knowledge 的事实来源，也不会绑定到其他对象；candidate 只从 source 生成，note 内容不能直接转化为 candidate。
- `deprecated` 表示 note 过时或不再推荐使用，不做物理删除。

## 7. Index

用途：保存从对象文件派生出来的定位视图。

路径：`indexes/*`

常见索引：

- `manifest.yml`
- `source-index.md`
- `knowledge-index.md`
- `tag-index.md`
- `kb-index.md`
- `knowledgebase/*-knowledge-index.md`
- `note-index.md`
- `review-queue.md`

规则：

- 索引不是事实来源。
- 索引可由对象文件重建。
- 索引损坏时以对象文件为准。
- 索引不得包含对象文件中不存在的事实。
- 默认列表索引只展示当前可用对象，`deprecated` 对象从列表索引中隐藏；对象文件仍是审计事实来源。

## 8. System Prompt

用途：描述 LLM 参与命令和 API 时使用的 KBManager 系统提示词。用户侧不提供提示词文件。

KBManager 系统提示词：

- 路径：KBManager 本体中的 `system-prompts/*.md`。
- 归属：随 KBManager 发布，不属于用户工作区。
- 内容：角色定义、边界、行为规则、输出约束、review gate、错误处理。
- 限制：不得包含用户 source、note、knowledge、索引或私有输入。

模板：

```md
---
id: system-prompt-candidate-create-v1
type: system-prompt
title: Candidate Create Prompt
api: kb.candidate.create
version: 1
inputs:
  - source
outputs:
  - candidate_draft_list
review_required: false
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---

## Role

## Input Variables

## Output Format

## Constraints

## Review Boundary
```

状态：

- `active`
- `deprecated`

关系：

- 系统提示词可被 Interface 命令和 `kb.*` API 的 `needs_llm` 请求引用。
- 被 API 运行记录使用版本。
- KBManager 系统提示词不存放具体业务输入。

## 9. 关系字段

当前对象模型中，知识组织、归属和结构位置由 knowledge base outlines YAML 与 knowledge `bindto` 表达；来源追溯由 `evidence` 表达。
