# 分阶段开发计划

本文基于 `docs/架构设计.md`、`docs/API设计.md`、`docs/Interface.md` 和 `docs/对象.md` 制定。开发遵循模块化、从小到大、先可验证后扩展的原则。

## 1. 开发原则

- 先实现第三层 Repository / Data Layer，再实现第二层 Application API / Domain Service，最后实现第一层 Interface / slash command。
- 任何对象写入、状态变化、文件移动和索引重建都必须通过 `kb.*` API。
- 索引只作为派生视图，测试中必须能通过对象文件重建索引。
- LLM 不直接写对象文件；涉及 LLM 的 API 先实现 `needs_llm` / `resume` 契约和校验，再接入真实编排。
- review gate 使用 Claude Code 对话中的确认、reviewed Markdown 或结构化参数表达。
- 每阶段必须有单元测试；跨模块能力进入集成测试；slash command 和用户流程进入端到端测试。

## 2. 阶段 0：项目骨架与测试基础

目标：建立可持续开发的工程骨架。

开发内容：

- 选择运行时与包结构，例如 `src/`、`tests/`、`system-prompts/`。
- 定义统一响应结构：`success`、`failed`、`needs_llm`、`needs_review`、`partial`。
- 建立错误类型、结果对象、路径配置、工作区上下文。
- 建立测试临时工作区工具，所有测试在临时目录中运行。
- 配置 lint、format 和测试命令。

验收标准：

- 可以运行空测试套件。
- 测试工具能创建和清理隔离工作区。
- 所有 API 返回值都能序列化为稳定结构。

测试重点：

- 响应结构序列化。
- 错误结构包含 operation、code、message、suggestion。
- 工作区路径不能逃逸到目标目录外。

## 3. 阶段 1：Repository / Data Layer 最小闭环

目标：先完成文件型事实来源的可靠读写。

开发内容：

- Markdown frontmatter 解析和写回。
- `.meta.yml` 解析和写回。
- 对象路径解析：source、candidate、knowledge、knowledge base、note、index。
- ID 扫描和唯一性检查。
- 对象序列化和反序列化。
- 原子写入、文件移动、目录创建。
- 非 Markdown source 的旁置 metadata 支持。

验收标准：

- 能读写一个 Markdown 对象，保持 frontmatter 和正文结构。
- 能读写一个 PDF source 的 `.meta.yml`。
- 能扫描工作区并发现重复 ID。
- 写入不会覆盖非预期用户文件。

测试重点：

- frontmatter 缺失、字段缺失、YAML 非法。
- Markdown 对象不得同时使用旁置 `.meta.yml`。
- PDF source 必须有同名 `.meta.yml`。
- 文件移动后源路径和目标路径状态正确。
- 路径穿越输入必须失败。

## 4. 阶段 2：`kb.init` 与模板系统

目标：交付第一个可运行 API，初始化受控工作区。

开发内容：

- 实现 `kb.init`。
- 创建目录结构、模板文件和空索引占位文件。
- 实现 dry-run。
- 实现冲突检测和幂等执行。

验收标准：

- 空目录执行 `kb.init` 后得到设计文档中的目录结构。
- 重复执行不修改已有兼容文件。
- 遇到同名非预期文件时整体失败，且不产生部分写入。

测试重点：

- dry-run 不写入。
- 幂等初始化。
- 冲突时零写入。
- 初始化产物不包含用户知识数据。

## 5. 阶段 3：Source 与 Candidate 最小链路

目标：打通 raw source 到 pending candidate 的候选知识链路。

开发内容：

- 实现 `kb.source.add` 的入参校验和 `needs_llm` 返回。
- 实现 `kb.source.add` resume，写入 source、cleaned 派生内容和元数据。
- 实现 `kb.candidate.create` 的 `needs_llm` 返回。
- 实现 `kb.candidate.create` resume，写入 pending candidate。
- 实现 source / candidate 引用校验和 evidence 校验。
- 实现 `kb.candidate.get`、`kb.candidate.next_pending`。

验收标准：

- 添加 Markdown source 后能生成 source 对象和 cleaned 文件。
- 基于 source 能生成一个或多个 pending candidate。
- candidate 使用全局 knowledge ID，但仍位于 `candidates/pending/`。
- LLM 输出结构不合法时 API 拒绝写入。

测试重点：

- `needs_llm` 阶段不得写入。
- resume token 错误必须失败。
- candidate 必须至少引用 source 或 note 之一。
- evidence 引用不存在时失败。
- 同一 ID 不得同时出现在 candidate 和 knowledge 中。

## 6. 阶段 4：Review Gate 与 Knowledge 状态流转

目标：实现人工 review 后的正式知识准入。

开发内容：

- 实现 `kb.knowledge.accept`。
- 实现 `kb.knowledge.reject`。
- 实现 `kb.candidate.defer`。
- 实现 `kb.knowledge.merge`。
- 实现 `kb.knowledge.deprecate`。
- 实现状态机校验、review 参数校验、对象移动和 review 记录。

验收标准：

- pending candidate 经 user review 后可提升为 accepted knowledge。
- accept 后同 ID candidate 文件不再保留。
- reject / defer 会移动 candidate 并写入 review 记录。
- merge 仅支持将 pending candidate 合入已有 knowledge；最终使用目标 knowledge ID，来源 candidate 被 rejected。
- 缺少 review 决策时返回 `needs_review`，不得写入。

测试重点：

- 缺少 user review 的 accept / reject / defer / merge / deprecate 都不能写入。
- 非 pending candidate 不能 accept。
- deprecated knowledge 保留引用链。
- merge 后关系、tags、kb_ids 正确。
- 状态转换非法时返回结构化错误。

## 7. 阶段 5：Knowledge Base 与 Note

目标：补齐用户组织知识和记录笔记的基础能力。

开发内容：

- 实现 `kb.knowledgebase.create`。
- 实现 `kb.note.add`、`kb.note.get`、`kb.note.deprecate`。
- 支持 note 与 source、candidate、knowledge、knowledge base 的绑定引用。
- 保持 knowledge base 成员维护边界：不提供 add/remove 成员 API。

验收标准：

- 用户 review 后可创建 knowledge base。
- note 可加入 inbox、读取、标记 deprecated 并移动目录。
- deprecated note 不物理删除。

测试重点：

- knowledgebase create 必须有 user approve。
- 重名或重复 ID 处理。
- note deprecate 必须记录原因。
- note 绑定对象不存在时失败或给出明确警告，按最终规则固定。

## 8. 阶段 6：索引重建与一致性检查

目标：让所有派生索引可重建、可 diff、可审计。

开发内容：

- 实现 `kb.index.rebuild`。
- 生成 `source-index.md`、`knowledge-index.md`、`tag-index.md`、`relation-index.yml`、`kb-index.md`、`note-index.md`、`review-queue.md`。
- 支持 `indexes/knowledgebase/<knowledgebase-id>-knowledge-index.md`。
- 实现 dry-run diff 和一致性问题报告。
- 实现对象文件优先于索引的冲突处理。

验收标准：

- 从对象文件可完整重建索引。
- dry-run 只返回拟更新 diff，不写文件。
- 重建索引不得反向修改对象 frontmatter。
- `kb.index.rebuild` 的 dry-run 输出可支撑问题清单和差异验证。

测试重点：

- 索引损坏后可重建。
- 对象与索引冲突时以对象为准。
- 缺失引用、重复 ID、candidate / knowledge 同 ID 共存可被发现。
- deprecated source / knowledge 默认从列表索引中隐藏，但对象文件和引用链必须保留，可通过 `/check` 发现相关一致性问题。

## 9. 阶段 7：Interface 与 Slash Command

目标：在 API 稳定后实现用户入口。

开发内容：

- 实现 `/init`。
- 实现 `/source add <path>`：编排 source add 和 candidate create。
- 实现 `/candidate review [candidate-id]`：get、review 辅助、Claude Code reviewed Markdown、accept / reject / defer / merge。
- 实现 `/knowledgebase create`、`/knowledgebase list`。
- 实现 `/note add/list/view/deprecate`。
- 实现 `/check`。

验收标准：

- slash command 不直接修改对象文件。
- 涉及写入的命令只调用第二层 API。
- `/check` 调用 `kb.index.rebuild` 重建索引并报告一致性问题。
- review 命令在用户未确认前不调用状态变更 API。

测试重点：

- 使用 mock API 验证命令编排顺序。
- `/source add` 正确处理两次 `needs_llm` 和 resume。
- `/candidate review` 四个分支都调用正确 API。
- `/check` 会重建索引并返回结果，不直接写对象文件。

## 10. 阶段 8：LLM Prompt 与编排校验

目标：把 LLM 纳入可控、可测试的边界。

开发内容：

- 创建 KBManager 自带 `system-prompts/*.md`。
- 实现 prompt 选择和组装顺序：系统提示词 -> 用户输入 -> 对象上下文 -> 输出 schema。
- 实现 LLM 输出 schema 校验。
- 实现 candidate review assist、knowledge merge assist、knowledgebase create 草案生成。
- 记录 prompt 版本到 API 运行记录中。

验收标准：

- 用户侧不提供提示词文件。
- prompt 不注入完整知识库正文，除非当前操作明确需要对象正文。
- LLM 输出不能绕过 review gate。
- LLM 生成事实必须携带 evidence 引用。

测试重点：

- prompt 组装顺序稳定。
- 用户输入不能覆盖系统约束。
- schema 不合法时拒绝 resume。
- LLM 生成 accepted knowledge 时必须失败。
- 缺少 evidence 的 candidate draft 必须失败。

## 11. 阶段 9：端到端验收场景

目标：以真实用户流程验证系统整体一致性。

端到端场景：

- 初始化工作区：`/init`。
- 导入 Markdown source：`/source add`。
- 生成 pending candidate。
- review accept，得到 accepted knowledge。
- 创建 knowledge base。
- 重建索引并运行 `/check`。
- 添加 note 并 deprecated。

验收标准：

- 全流程不需要数据库、向量库、全文搜索服务。
- 所有业务事实都来自 Markdown、PDF、YAML。
- 所有正式 knowledge 都经过 review。
- 删除语义均为 deprecated 或 archived，不物理删除对象。
- 索引可以删除后重建。

测试重点：

- 端到端测试使用临时工作区和 mock LLM。
- 对象文件快照测试。
- 索引快照测试。
- review gate 回归测试。

## 13. 推荐里程碑

| 里程碑 | 范围 | 主要产物 | 测试门槛 |
| --- | --- | --- | --- |
| M1 | 阶段 0-2 | 工程骨架、数据层、`kb.init` | 单元测试通过，初始化幂等 |
| M2 | 阶段 3 | source 到 candidate | mock LLM 集成测试通过 |
| M3 | 阶段 4 | review 到 knowledge | 状态机和 review gate 测试通过 |
| M4 | 阶段 5-6 | knowledgebase、note、index | 索引重建和一致性测试通过 |
| M5 | 阶段 7-8 | slash command、prompt 编排 | 命令编排测试通过 |
| M6 | 阶段 9 | 端到端验收 | 临时工作区 E2E 测试通过 |

## 14. 优先级建议

第一优先级：

- `kb.init`
- frontmatter / `.meta.yml` 数据层
- ID 唯一性和路径安全
- source -> candidate -> review -> knowledge
- review gate

第二优先级：

- 索引重建
- knowledgebase create
- note add / get / deprecate
- `/check`

第三优先级：

- 更完整的 Claude Code review 交互
- 更丰富的 prompt 版本记录和审计报告
