# 架构设计

## 1. 设计立场

本系统坚持：

- 不使用 RAG。
- 不使用数据库。
- 不使用向量库。
- 不使用全文搜索服务。
- 所有持久化数据均为 Markdown、PDF、YAML。
- 允许维护文件型索引，但索引只是派生视图，不是事实来源。

系统目标不是构建企业级知识平台，而是构建一个面向个人使用的、可审计、可迁移、可版本管理、可由 Claude Code 通过 API 编排操作的文件型 AI 知识中台。

KBManager 本身是中台程序和规则集合，不包含用户知识数据。用户使用时会在自己的 git 工作区中运行 KBManager，该工作区内的 source、candidate、knowledge、knowledge base、note、索引和输入内容都属于用户。

## 2. 核心原则

### 2.1 文件即系统

所有核心数据都是文件：

- 原始数据：`md` / `pdf`
- 元数据：Markdown 文件使用 YAML frontmatter；非 Markdown 文件使用同名 `.meta.yml`
- 候选知识：`md`
- 正式知识：`md`
- 知识库：`md`
- 笔记：`md`
- 索引：`md` / `yml`
- Review 队列：`md`
- KBManager 系统提示词：随中台发布，用于定义角色、边界、行为和输出约束。用户侧不提供提示词文件。

不得引入数据库、RAG 服务、向量检索、ElasticSearch、SQLite、Postgres、Chroma、FAISS 等作为系统事实来源。

### 2.2 索引不是数据库

索引文件只用于辅助人、Claude Code 和 API 快速定位文件，不承担数据库语义。

索引文件必须满足：

- 可读。
- 可编辑。
- 可 diff。
- 可手工修复。
- 可由一致性检查流程重建。
- 不作为唯一事实来源。

对象事实来源永远是用户工作区中的具体 source、candidate、knowledge、knowledge base、note 文件；其中 note 只作为 note 对象自身的事实来源，不作为 knowledge 的 evidence 或 candidate 上游。KBManager 自带系统提示词不是用户知识事实来源。

### 2.3 元数据统一规则

Markdown 文件的元数据必须写在 YAML frontmatter 中。

```md
---
id: knowledge-20260520-001
type: knowledge
title: 示例知识
status: accepted
tags:
  - ai
  - knowledge-management
created: 2026-05-20T14:30:05
updated: 2026-05-20T14:30:05
---
```

非 Markdown 文件的元数据写在同名 `.meta.yml` 中，例如：

```txt
data/raw/pdf/example.pdf
data/raw/pdf/example.meta.yml
```

同一个资源只能有一个元数据事实来源。Markdown 文件不得同时维护 YAML frontmatter 和旁置 `.meta.yml`。索引文件从 YAML frontmatter 或 `.meta.yml` 派生，不反向作为事实来源。

### 2.4 知识必须可引用

知识不是一篇文章，也不是一段摘录，而是一个独立的、可被引用的认知单元。

一个知识单元必须包含：

- 唯一 ID。
- 标题。
- 知识正文。
- 支撑材料。
- 来源。
- 所属知识库。
- 状态。
- 创建与更新时间。

### 2.5 人类 Review 是准入门槛

LLM 不能直接把候选内容写入正式知识库。

LLM 只能生成：

- 摘要。
- 清洗结果。
- 知识生成建议。
- 合并建议。
- 冲突提示。
- 候选知识。
- 索引重建草案。

正式知识的创建、合并、拒绝、拆分、废弃必须经过 user review。

## 3. 分层架构

系统分为三层：

```txt
Layer 1: Interaction & Orchestration
  Claude Code UI、kbm-* skills、自然语言交互、结果展示、人工 review

Layer 2: Application API / Domain Service
  原子 API、组合 API、状态机、权限校验、review gate、一致性规则

Layer 3: Repository / Data Layer
  Markdown/PDF/YAML 文件读写、frontmatter、索引派生、路径管理、对象序列化
```

### 3.1 第一层：交互与编排层

第一层负责用户输入和结果展示。

输入方式包括：

- 用户自然语言指令。
- 用户 review、确认、拒绝、选择、补充原因。

第一层可以做：

- 理解用户意图。
- 把自然语言转换为一个或多个 API 调用。
- 编排 API 调用顺序。
- 展示 API 返回的结果、diff、错误、review 选项。
- 在需要 user review 的步骤暂停。

第一层不得：

- 绕过 API 直接修改对象文件；clean 迁移和 `kbm-kb` 明确触发的 outline YAML 更新除外。
- 自行决定正式 knowledge 入库、合并、拆分、废弃。
- 把索引当作事实来源。
- 直接维护对象状态机。

第一层可以为展示、问答、review 和 check workflow 只读读取对象文件和索引文件。只读读取不得改变对象状态，不得把索引内容反向写入对象事实。任何对象写入、状态变化、文件移动和索引重建都必须通过第二层 API。

### 3.2 第二层：Application API / Domain Service

第二层是系统核心。它对第一层提供可编排的原子 API 和少量组合 API。

第二层负责：

- 定义所有可执行能力。
- 校验对象是否存在。
- 校验状态转换是否合法。
- 提供可由 Claude Code PreToolUse hook 审批的最终写入 API，并在 API 内校验 reviewed payload。
- 分配 ID。
- 更新对象元数据。
- 调用数据层读写文件。
- 决定哪些索引需要重建。
- 返回结构化结果、错误和影响范围。

第二层不依赖 Claude Code 的对话上下文。同一个 API 可以被内部 JSON helper、MCP server、脚本或测试调用。详细 API 设计见 `docs/API设计.md`，API 流程见 `docs/流程图.md`。

原子 API 示例：

```txt
kb.init
kb.source.add
kb.source.deprecate
kb.candidate.create
kb.candidate.defer
kb.knowledge.accept
kb.knowledge.merge
kb.knowledge.reject
kb.knowledge.deprecate
kb.knowledgebase.create
kb.note.add
kb.note.deprecate
kb.index.rebuild
```

第一层可以根据意图调用多个 API，但不得把多个状态变化塞进一个不可审计的黑盒操作。

### 3.3 第三层：Repository / Data Layer

第三层只负责数据存取和序列化。

第三层负责：

- Markdown frontmatter 读写。
- `.meta.yml` 读写。
- PDF、Markdown、附件路径管理。
- 对象序列化和反序列化。
- 文件移动和归档存放。
- 索引文件生成和覆盖。
- 基础唯一性扫描。

第三层不得：

- 决定正式知识是否入库。
- 决定 review 是否通过。
- 决定状态转换是否合法。
- 把索引内容反向写回对象事实。

除 clean 特许迁移执行和 `kbm-kb` 明确触发的 outline YAML 更新外，所有对数据层的修改必须通过第二层 API。Claude Code UI 和 `kbm-*` skills 可以只读读取用户工作区文件用于展示、问答和检查，但不能直接改数据层。

## 4. 数据生命周期

核心流转路径：

```txt
source
  -> source.cleaned 派生字段
  -> candidate create 读取已有 source IDs
  -> candidate
  -> user review
  -> knowledge
  -> knowledge base
  -> derived indexes
```

- `source`：原始材料，保持尽量接近导入时状态；状态为 `raw`、`linked` 或 `deprecated`。
- `source.cleaned`：source 对象中的清洗派生字段，可指向 `data/cleaned/` 下的可读版本，仍然不是正式知识。
- `candidate`：通过 `kb.candidate.create` 从已有 source IDs 生成的候选知识，使用全局 knowledge ID。
- `user review`：用户确认候选是否接受、拒绝、延后、合并或拆分。
- `knowledge`：通过 review 的正式知识单元。
- `knowledge base`：面向主题或用途组织 knowledge 的集合，由 knowledgebase create workflow 从 source-like input 生成并经用户 review 后创建 active 对象；`description`、`scope`、`default_outline_id` 和 outline manifest 存在 knowledgebase Markdown 中，完整 outline 树存在同名 outlines YAML 中；knowledge 归属以 knowledge 对象上的 `bindto` 为事实来源，知识库成员视图由索引派生。
- `derived indexes`：从对象元数据重建出来的定位视图。

任何阶段的派生文件都必须能追溯到上游来源。无法追溯来源的内容只能作为 note，不能直接进入正式 knowledge。

├── README.md

│   ├── raw/

## 6. 目录职责

- `data/raw/`：保存原始 source 和对应元数据。
- `data/cleaned/`：保存清洗后的 source 派生版本。
- `data/attachments/`：保存图片、附件、辅助材料。
- `candidates/`：保存候选知识及其 review 状态。
- `knowledge/atomic/`：保存正式原子知识。
- `knowledge/bases/`：保存知识库定义和知识集合。
- `notes/`：保存个人笔记。note 默认不是正式知识。
- `indexes/`：保存派生索引和 review 队列。

对象模板是 KBManager 自带系统资源，随发布包维护，不属于用户工作区数据层。

## 7. LLM 参与方式

LLM 参与系统时必须遵循 API 边界和最小写入原则：

- 可以通过第一层接收用户输入并生成 API 编排计划。
- 可以通过第二层 API 获取用户工作区中的 source、candidate、knowledge、knowledge base、note 和 index；note 只用于 note 操作和只读展示，不作为 candidate 生成上游。
- 可以接管第二层 API 返回的 `needs_llm` 请求，生成 source `summary`、`tags` 和 cleaned 派生内容、candidate、knowledgebase draft、已有 knowledgebase 的归属建议、标题摘要和 clean migration plan 等中间结果。
- 可以请求第二层 API 根据对象元数据重建索引。
- 不得绕过 user review 直接创建 accepted knowledge。
- 不得绕过第二层 API 直接改对象文件或索引文件；clean 迁移和 `kbm-kb` 明确触发的 outline YAML 更新除外。
- 不得把索引文件当成事实来源。
- 不得要求或读取用户侧提示词文件。
- 不得把用户数据写入 KBManager 自带目录。

LLM 调用时的 prompt 只包含：

- KBManager 系统提示词：由中台提供，覆盖角色定义、操作边界、行为规则、输出格式、错误处理和安全约束。
- 用户输入和对象上下文：来自本次命令参数、自然语言、Claude Code reviewed content、review 备注，以及 API 提供的对象内容或摘要。

组装顺序必须是：KBManager 系统提示词 -> 本次用户输入 -> 对象上下文 -> 输出 schema。用户输入和对象上下文不能覆盖系统提示词中的安全边界、review gate、数据层写入规则和事实来源规则。

命令执行时由第一层或第二层引用对应系统提示词，并通过 API 将本次运行输入、输出和决策记录写入用户工作区对象文件。

第二层 API 不直接调用 LLM。需要 LLM 时必须返回 `needs_llm`，由 Claude Code 执行 LLM 调用，再通过 `resume_token` 将结果交回 API 校验和落盘。

## 8. 一致性规则

- ID 必须全局唯一。
- 每个对象必须声明 `id`、`type`、`created`、`updated`。
- 引用其他对象时必须使用对象 ID，不使用相对路径作为主引用。
- 路径可以作为辅助定位信息，但 ID 才是逻辑引用。
- 所有写入必须通过 Application API / Domain Service 层。
- KBManager 本体目录不得保存用户 source、note、knowledge 或索引。
- 用户工作区是用户知识数据的唯一持久化位置。
- 索引损坏时以对象文件重建。
- 对象文件和索引冲突时，以对象文件为准。
- user review 记录必须能追溯到 candidate。
- `deprecated` 表示内容过期或不再推荐使用。
- deprecated source 或 knowledge 必须保留对象文件和引用链；默认列表索引隐藏 deprecated 对象，review 流程和 check workflow 应在相关引用出现时提示过期状态。
